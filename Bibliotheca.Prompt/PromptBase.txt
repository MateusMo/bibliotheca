Bibliotheca — Especificação Geral do Projeto
1. Introdução
1.1 Visão Geral

A Bibliotheca é uma plataforma colaborativa para catalogação de bibliotecas pessoais, livros raros, edições históricas e exemplares físicos, inspirada no conceito do Discogs.

Diferentemente de plataformas tradicionais de leitura, a Bibliotheca possui como principal objetivo preservar e documentar a história de cada exemplar físico, permitindo que colecionadores, pesquisadores, bibliotecas, sebos e leitores registrem suas coleções de forma detalhada.

O sistema não trata apenas o livro como uma obra intelectual, mas também como um objeto histórico.

Cada exemplar possui sua própria identidade, podendo conter características únicas como edição, estado de conservação, autógrafos, ex-líbris, dedicatórias, fotos, local de publicação e diversas outras informações relevantes para colecionadores.

1.2 Objetivo

Construir a maior plataforma colaborativa em língua portuguesa voltada para:

catalogação de bibliotecas pessoais;
preservação da história editorial;
documentação de livros raros;
organização de coleções particulares;
descoberta de novas bibliotecas;
formação de uma comunidade de colecionadores.

A plataforma deverá permitir que qualquer usuário explore bibliotecas públicas, descubra obras históricas e acompanhe estatísticas detalhadas das coleções.

1.3 Público-alvo

A plataforma é destinada principalmente a:

Bibliófilos.

Universidades.

3. Requisitos Não Funcionais
RNF-01 — Arquitetura

O projeto deverá seguir Clean Architecture.

RNF-02 — Escalabilidade

O sistema deverá suportar milhares de usuários simultaneamente.

Toda listagem deverá utilizar paginação.

RNF-03 — Segurança

As senhas deverão utilizar BCrypt.

Jamais poderão ser armazenadas em texto puro.

RNF-04 — Autorização

Toda operação de escrita deverá validar o proprietário do recurso.

RNF-05 — Performance

Consultas deverão utilizar paginação.

Operações assíncronas deverão utilizar async/await.

RNF-06 — Persistência

Banco MySQL utilizando EF Core.

RNF-07 — Soft Delete

Nenhuma entidade será removida fisicamente.

Toda exclusão utilizará:

IsActive = false
RNF-08 — Concorrência

Cada requisição utilizará seu próprio DbContext.

RNF-09 — Padronização

Todos os Services retornarão:

ResponseDto<T>
RNF-10 — Código

Todo acesso ao banco ocorrerá exclusivamente através de Repositories.

A camada Web nunca acessará diretamente o DbContext.

RNF-11 — Testabilidade

Toda dependência deverá possuir interface.

RNF-12 — Organização

Nenhuma camada poderá violar as dependências da Clean Architecture.

5. Objetivos Arquiteturais
Objetivo 1 — Suporte a múltiplos usuários

Cada usuário possui sua própria biblioteca.

Todo recurso pertence exclusivamente ao seu proprietário.

Consultas deverão ser isoladas por UserId.

Toda edição deverá validar a posse do recurso no servidor.

Objetivo 2 — Segurança
BCrypt
Cookie Authentication
DataAnnotations
Soft Delete
SQL parametrizado
DTOs sem exposição de senha
Verificação de autorização em todas as operações de escrita
Objetivo 3 — Manutenibilidade
Separação rigorosa entre camadas
Repositories especializados
Unit of Work
Generic Repository
DTOs
Services
Interfaces
ResponseDto padronizado
Convenção uniforme de nomenclatura
Filtros encapsulados em objetos próprios
